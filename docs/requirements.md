# ex-g-pro-remapper 要件定義書

## 1. 目的

本システムは、Linux 上で物理マウスから入力されるイベントを監視し、定義済みのルールに従ってキーボード入力へ変換する常駐デーモンである。

出力は仮想マウスではなく仮想キーボードとして OS に提示し、systemd によって起動・停止・監視されることを前提とする。

## 2. スコープ

### 2.1 対象

- 物理マウスのボタン入力を契機とするキーボード入力へのリマップ
- 設定ファイルによるリマップルール定義
- systemd 管理下での常駐実行
- Linux の `evdev` / `uinput` を利用した入力監視と仮想キーボード生成

### 2.2 非対象

- 仮想マウスデバイスの生成
- GUI 設定ツール
- 動的なクラウド連携
- Wayland/X11 固有 API に依存した実装
- 高度なマクロ編集 UI

## 3. 想定利用環境

- OS: systemd を採用する Linux ディストリビューション
- 入力取得: `/dev/input/event*`
- 仮想デバイス生成: `/dev/uinput`
- 実行形態: systemd service として常駐
- 実装言語: Rust

## 4. 用語定義

- 物理入力デバイス: 実機のマウス
- 仮想キーボード: `uinput` により生成されるキーボードデバイス
- リマップルール: 物理マウス入力を仮想キーボード入力へ変換する定義
- 設定ファイル: YAML または独自文法で記述されるルールファイル

## 5. システム概要

本システムは以下の処理を行う。

1. 対象マウスデバイスを特定する
2. 設定ファイルを読み込み、リマップルールを構築する
3. `uinput` により仮想キーボードを生成する
4. マウスイベントを監視する
5. 一致するルールが存在する場合、対応するキーボードイベントを仮想キーボードから送出する
6. systemd により異常終了時の再起動とログ収集を行う

## 6. 機能要件

### 6.1 デバイス監視

- FR-001: システムは 1 台の物理マウス入力デバイスを監視できること
- FR-002: 監視対象デバイスはデバイスパス、by-id パス、または識別子ベースで指定できること
- FR-003: 指定デバイスが存在しない場合、明確なエラーログを出力して終了または再試行できること
- FR-004: 同時に複数マウスを監視対象として扱わないこと

### 6.2 リマップ処理

- FR-010: システムはマウス入力を契機としてキーボードイベントへ変換できること
- FR-011: 少なくとも押下と解放の両イベントを扱えること
- FR-012: 1 つのマウス入力に対して 1 つ以上のキーボードイベント列を出力できること
- FR-013: 未定義の入力は無視できること
- FR-014: 将来的な拡張を考慮し、単発キー送出以外に複数キー送出へ対応可能な内部構造とすること
- FR-015: 初期実装で扱う入力イベントはマウスボタンイベントに限定すること
- FR-016: ホイールイベントおよび移動イベントは初期実装では扱わないこと
- FR-017: 同一入力条件に対して複数ルールが競合する場合、設定ファイル内で後に記述されたルールを優先すること
- FR-018: 競合ルールを検出した場合、起動時または設定再読込時に警告ログを出力すること

### 6.3 設定ファイル

- FR-020: リマップ定義は外部設定ファイルから読み込めること
- FR-021: 設定フォーマットは YAML または独自文法のいずれかを採用できること
- FR-022: 初期実装では YAML を優先し、独自文法は将来拡張としてもよい
- FR-023: 設定ファイルの構文エラー時は、行番号または該当ルールを含む診断情報を出力すること
- FR-024: 設定ファイルには少なくとも以下を記述できること
- FR-025: 対象デバイスの指定
- FR-026: 入力条件
- FR-027: 出力するキーコード
- FR-028: 押下時と解放時の動作
- FR-029: 設定ファイルパスはコマンドライン引数または固定パスで指定できること
- FR-030: 設定ファイルの変更を検知し、プロセス再起動なしでルールを再読込できること
- FR-031: 再読込失敗時は直前に有効だった設定を保持し、エラーログを出力すること

### 6.4 仮想キーボード

- FR-040: システムは仮想キーボードを生成すること
- FR-041: 仮想デバイスは OS からキーボードとして認識されること
- FR-042: 仮想デバイスには設定で使用される全キーコードを登録できること
- FR-043: 仮想マウスデバイスは生成しないこと

### 6.5 デーモン運用

- FR-050: systemd service として起動可能であること
- FR-051: フォアグラウンド実行を前提とし、systemd にプロセス管理を委譲すること
- FR-052: systemd による自動再起動設定を可能とすること
- FR-053: 標準出力または標準エラーへログを出力し、journald で収集可能であること
- FR-054: 停止シグナルを受けた場合は安全に終了処理を行うこと
- FR-055: systemd service は root 権限で実行すること

## 7. 設定仕様要件

### 7.1 YAML 採用時の最低要件

- NFR-100: 人手で編集しやすいこと
- NFR-101: Rust の `serde` 系ライブラリで安定してパース可能であること
- NFR-102: コメント記述または可読性を確保できること

### 7.2 独自文法採用時の最低要件

- NFR-110: YAML より高い表現力または簡潔性があること
- NFR-111: パーサ実装コストと保守コストを明確化できること
- NFR-112: 構文エラー時の診断品質を担保できること

### 7.3 推奨方針

- 初期リリースは YAML を採用する
- 独自文法は、複雑なマクロ、条件分岐、レイヤー切替などが必要になった場合に再検討する

## 8. 非機能要件

### 8.1 性能

- NFR-001: 入力イベント検知から仮想キーボード送出までの遅延は、通常利用で体感上問題ない水準であること
- NFR-002: 常駐時の CPU 使用率は低く抑えられること
- NFR-003: 不要なポーリングを避け、イベント駆動で処理すること

### 8.2 信頼性

- NFR-010: 設定不備、デバイス未接続、権限不足を識別可能な形でログ出力すること
- NFR-011: 異常終了時に systemd の再起動ポリシーと整合すること
- NFR-012: 仮想デバイス生成失敗時は中途半端な状態を残さず終了すること
- NFR-013: ホットリロード中に一時的な設定更新失敗が発生しても、入力変換処理を継続できること

### 8.3 保守性

- NFR-020: 入力取得、設定パース、リマップ判定、仮想デバイス出力をモジュール分離できる構造とすること
- NFR-021: ルール追加時に既存処理への影響を局所化できること
- NFR-022: ログメッセージは障害解析に必要な識別情報を含むこと

### 8.4 セキュリティ

- NFR-030: root 実行を前提としつつ、systemd の保護設定により不要な権限拡大を抑制すること
- NFR-031: 設定ファイルは信頼できる管理者のみが編集可能であること
- NFR-032: 任意コード実行につながる設定解釈は行わないこと

## 9. systemd 要件

### 9.1 サービス構成

- systemd unit は `Type=simple` を基本とする
- デーモン自身はバックグラウンド化しない
- `Restart=on-failure` を推奨する
- 設定ファイルパスや対象デバイス指定は `ExecStart` 引数または `EnvironmentFile` で渡せる設計とする
- `User=root` または root 実行相当の systemd 設定を前提とする
- 設定変更のホットリロードはプロセス内監視を基本とし、systemd の再起動に依存しない

### 9.2 運用要件

- サービス起動時に設定ファイル妥当性とデバイスアクセス可否を検証すること
- journald から起動失敗原因を追跡可能であること
- OS 起動時の自動起動を可能にすること
- 競合ルール検出時は起動継続しつつ警告ログを残すこと

## 10. 制約条件

- Linux の `evdev` と `uinput` が利用可能であること
- 実行ユーザーは root であること
- キー送出は Linux 入力サブシステムが扱えるキーコードに限定されること
- 初期実装ではボタンイベントのみを入力対象とすること

## 11. 想定設定例

```yaml
device:
  path: /dev/input/by-id/usb-Logitech_Gaming_Mouse-event-mouse

remaps:
  - input:
      type: key
      code: BTN_RIGHT
      value: 1
    output:
      - key: KEY_LEFTMETA
        value: 1

  - input:
      type: key
      code: BTN_RIGHT
      value: 0
    output:
      - key: KEY_LEFTMETA
        value: 0
```

## 12. 受け入れ条件

- AC-001: 指定した物理マウス入力を監視できること
- AC-002: 設定ファイルのルールに従い、仮想キーボードからキー送出できること
- AC-003: OS から仮想デバイスがキーボードとして見えること
- AC-004: 仮想マウスデバイスを生成しないこと
- AC-005: systemd service として起動、停止、再起動できること
- AC-006: 設定不正時に起動失敗と理由がログで確認できること
- AC-007: 設定ファイル変更時にプロセス再起動なしで新ルールへ切り替わること
- AC-008: 無効な再読込が発生しても既存ルールで動作継続すること
- AC-009: 競合ルールがある場合、後勝ちで解決され、かつ警告ログが残ること
- AC-010: ホイールおよび移動イベント入力ではリマップ処理を行わないこと

## 13. 実装方針の推奨

- 第 1 段階: YAML のみ対応して最小実装を成立させる
- 第 2 段階: 複数キー出力、設定検証強化、ホットリロードを行う
- 第 3 段階: 必要性が明確になった場合のみ独自文法を追加する

## 14. 決定事項

- 監視対象は単一マウスとする
- 初期実装ではホイールイベントおよび移動イベントは扱わない
- 設定変更のホットリロードを実装する
- systemd service は root 実行とする
- 競合ルールは後勝ちで解決し、同時に警告ログを出力する
